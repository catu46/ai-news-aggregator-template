# src/ — the project's Python package

This folder is all of the project's code: a personal AI news/repo aggregator. The idea is simple: sources (Reddit, GitHub, X) dump posts into a Postgres database (Supabase + pgvector), each post gets an embedding (Voyage) and a quality verdict from an LLM curator (Haiku 4.5), and whatever passes is delivered on Telegram with 👍/👎 buttons. The votes become a taste signal and a meaning-searchable archive. Here at the package root live only the two data-side "entry scripts" (`pipeline.py` and `seed.py`); each subfolder is a piece of the system with its own CLAUDE.md (read the subfolder's before touching it). Code comments are in English, identifiers in English.

## Files at this root

- `pipeline.py` -> Orchestrates **one complete cycle** of data: `ingest -> embed -> curate`, in this order (each step feeds the next). Runs with `python -m src.pipeline` and is built to run on a schedule (e.g.: cron on Railway): it's idempotent, each step only touches what's pending in the database, and deduplication by `(source_platform, source_id)` happens on upsert. **Ingest** (`run_ingestion`) builds the sources per user from `config/sources.yaml` and injects the active `/focus` topics straight into COLLECTION — they don't just reorder delivery in the bot, they actually PULL in new content. Per user, it reads `db.active_focus(user_id, "repos")` and `db.active_focus(user_id, "news")` and routes the topics by destination: **news** becomes a search on **Reddit** (`searches=` of `RedditSource`, added to the fixed `subreddits`) and on **X** (added to the accounts/`x_searches`); **repos** becomes a query on **GitHub** (added to the `github_queries`). A source is only built if it has some target (fixed subs/accounts/queries or focus topics). **Embed** (`run_embedding`) vectorizes posts without an embedding in batches. **Curate** (`run_curation`) asks Haiku for a GLOBAL quality verdict (`similarity_signal=None`, no user signal) and passes the owner's active interests — `db.all_active_focus_topics()` (the topics they want RIGHT NOW) — to the curator via `interests=`, loosening the bar for those topics; it respects the monthly spending cap (`BudgetExceeded` ends the step gracefully). Failures from a source/post are isolated and logged, they don't bring down the cycle. **Important:** delivery to Telegram does NOT happen here — it's the bot's JobQueue that delivers.

- `seed.py` -> **Cold-start:** loads taste examples from `config/seeds.yaml` BEFORE any real ingestion, so the system already has signal on day 1. Runs with `python -m src.seed`. For each example, it creates a post with `source_platform='seed'` and a deterministic `source_id` (`seed-<sha1>`, so re-running doesn't duplicate), embeds the text, and writes a preloaded vote: `gold => +1` (👍), `noise => -1` (👎), with `origin='seed'`. These votes+embeddings feed RECALL and the affinity prior, but the seeds **do not go through the curator** (the verdict is left NULL on purpose — they're taste labels, not approved/rejected posts) and don't enter the delivery flow. (The CHECK on `posts.source_platform` in `db/schema.sql` already includes `'seed'`, `'github'` and `'manual'`, so there's no extra schema step.)

## Subpackages (each has its own CLAUDE.md)

- `common/` -> The core shared by EVERYTHING else. `config.py` (reads `.env` + `config/sources.yaml` + `config/seeds.yaml` into typed objects, e.g.: `Settings`, `UserSources`, `SeedExample`), `db.py` (async access to Postgres/Supabase with pgvector via asyncpg — `upsert_post`, `set_embedding`, `mark_curation`, `record_vote`, `active_focus`, semantic search, etc.), `embeddings.py` (Voyage `voyage-4-lite` wrapper, 1024-dim L2-normalized vectors), `models.py` (shared data models: `IngestedPost`, `Verdict`). See `common/CLAUDE.md`.

- `ingestion/` -> The content **sources**. Every source inherits from `IngestionSource` (`base.py`) and implements `async fetch() -> list[IngestedPost]`; dedup is left to the database. Implementations: `reddit_source.py` (public RSS feed with multireddit), `github_source.py` (trending repos by topic via Search API + README), `x_source.py` (X/Twitter via `twitter-cli` with cookies). `pipeline.run_ingestion` is what instantiates and runs these sources. See `ingestion/CLAUDE.md`.

- `curation/` -> The **curator** and chat-based steering. `curator.py` (the `Curator` ABC interface + the default `AnthropicCurator` using Haiku 4.5 with Structured Outputs and the rubric in a cached system block; `BudgetExceeded` on the spending cap — it's "swappable", switching providers is just reimplementing `classify`), `prompt.py` (the long, static `RUBRIC` that triggers prompt caching), `steering.py` (classifies an owner's chat message into an intent: `steer` / `recall` / `balance` / `status` / `other`). See `curation/CLAUDE.md`.

- `bot/` -> The **Telegram bot** (`bot.py`, python-telegram-bot, long-polling). Locks onto an allowlist of `telegram_user_id`s coming from `sources.yaml` (multi-user-ready), delivers approved posts not yet delivered (`deliver_pending`, via JobQueue once a day in 2 buckets and on demand with `/feed`), records 👍/👎 votes from the inline buttons, routes free chat (steer/recall/balance/status) and runs semantic search over the archive (`/search`). State (Database + Embedder + allowlist) lives in `application.bot_data`. Run with `python -m src.bot.bot`. See `bot/CLAUDE.md`.

- `mcp_server/` -> An **MCP** server (`server.py`) that exposes YOUR curated archive as tools for Claude (Code/Desktop) — a thin shell over the `Database`, reusing what the bot already uses (`search_pool` / `recall_voted` / `active_focus`). It lets you ask "what did I like about RAG?" straight from Claude, without opening Telegram. Each person runs their own instance (their Supabase, their `telegram_user_id`), the same multi-tenant isolation as the rest. See `mcp_server/CLAUDE.md`.
