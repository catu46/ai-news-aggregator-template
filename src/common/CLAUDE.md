# src/common ‚Äî shared core

This folder is the "infrastructure" that everything else in the project uses: how to read configuration, how to talk to the database, what the data shapes flowing between modules are, and how to generate embeddings. No file here does product logic (ingestion, curation, bot) ‚Äî they just provide the pieces those modules call. If you're onboarding, start here: understanding these four files is understanding the vocabulary of the whole system.

Flow overview: the sources (Reddit/X/GitHub) produce `IngestedPost` ‚Üí they land in Postgres via `upsert_post` ‚Üí they get an embedding (Voyage) ‚Üí they go through curation (Haiku, produces `Verdict`) ‚Üí the approved ones are delivered to the user on Telegram, who votes üëç/üëé ‚Üí votes become a personal archive searchable by meaning (pgvector). The owner's chat messages become `ChatIntent`/`FocusItem` to steer the feed.

## File by file

**`config.py` ‚Äî typed configuration (`.env` + the YAMLs)**
- `Settings` (frozen dataclass): all the app's keys/secrets and parameters. Required (error if missing): `ANTHROPIC_API_KEY`, `VOYAGE_API_KEY`, `TELEGRAM_BOT_TOKEN`, `DATABASE_URL`. With defaults: `CURATOR_MODEL` (claude-haiku-4-5), `CURATOR_MONTHLY_BUDGET_USD` (8), `EMBEDDING_MODEL` (voyage-4-lite), `REDDIT_USER_AGENT`. Optional (may be None): Twitter tokens (`auth_token`/`ct0`), `GITHUB_TOKEN`.
- Curator provider (lets you swap the curation LLM without touching code): `CURATOR_PROVIDER` (`anthropic`, default, or `kimi`); when `kimi`, it uses `MOONSHOT_API_KEY` (optional, may be None), `MOONSHOT_BASE_URL` (default `https://api.moonshot.ai/v1`) and `KIMI_MODEL` (default `kimi-k2.6`). This is what decides which `Curator` implementation the rest of the app instantiates.
- Daily delivery: `DIGEST_HOUR` (local hour 0‚Äì23, default 7) and `DIGEST_TZ` (IANA timezone, default `America/Sao_Paulo`) ‚Äî they define the time of the digest scheduled by the bot.
- `load_settings()` reads the `.env` (via `load_dotenv()` at import) and builds the `Settings`. It's the single entry point for configuration ‚Äî other modules receive this object.
- `load_sources()` reads `config/sources.yaml` and returns a list of `UserSources` (one per user): where to fetch content for that user ‚Äî `subreddits`, X accounts and searches, GitHub queries ‚Äî tied to the `telegram_user_id`. It's what ingestion consumes.
- `load_seeds()` reads `config/seeds.yaml` and returns `{user_key: [SeedExample]}`. These are cold-start examples labeled `gold` or `noise` to give curation/affinity a reference before the user has any history. Ignores placeholders (empty text or starting with `<`). Links to the user by the same `key` used in `sources.yaml`.
- `ROOT`/`CONFIG_DIR` point to the project root and the `config/` folder (resolved from this file's path).

**`db.py` ‚Äî Postgres/pgvector layer (Supabase), everything async via `asyncpg`**
- `Database(dsn)` class: connection pool (1‚Äì5). `connect()`/`close()` manage the pool; `pool` (property) gives access and blows up if you forget to connect. In `_init_conn` it registers the `vector` type (pgvector) and the `jsonb` codec. `statement_cache_size=0` is on purpose ‚Äî compatibility with the Supabase pooler (pgbouncer/Supavisor in transaction mode).
- `get_or_create_user(telegram_user_id, display_name)` ‚Üí ensures the user row and returns the internal `id` (the rest of the API uses this `user_id`, not the Telegram one).
- `upsert_post(IngestedPost)` ‚Üí inserts a new post and returns the `id`; if it already existed (same `source_platform` + `source_id`), returns `None`. It's the ingestion dedup.
- `posts_pending_embedding(limit)` / `set_embedding(post_id, embedding, model)` ‚Üí queue and write for posts that don't have a vector yet. Pairs with `embeddings.py`.
- `posts_pending_curation(limit)` ‚Üí posts without a verdict yet, for the curator to process.
- `mark_curation(post_id, Verdict, model)` ‚Üí writes the curation result (approved/rejected, confidence, category, reject reason, summary, rationale). `mark_curation_error(...)` marks the verdict as `error` when the LLM call fails.
- `approved_undelivered(user_id, limit, max_age_days?)` ‚Üí approved posts NOT yet delivered to this user (anti-join with `deliveries`), ordered by `published_at DESC NULLS LAST` and already carrying `published_at` in the row. `max_age_days`: if set, discards posts published more than N days ago (keeps the ones WITHOUT a date) ‚Äî feed freshness. It's the delivery queue.
- `record_delivery(user_id, post_id, telegram_message_id, affinity_score)` ‚Üí records that a post was delivered (idempotent per user+post).
- `record_vote(user_id, post_id, vote, origin, telegram_message_id)` ‚Üí writes the vote (1 or -1); UPSERT, 1 vote per (user, post), re-voting overwrites.
- `vote_counts(user_id, platforms?)` ‚Üí `(likes, dislikes)` tuple, optionally filtering by platform.
- `nearest_votes(user_id, query_embedding, k, platforms?)` ‚Üí the k nearest ALREADY-VOTED posts to the given vector; filtering by platform keeps affinity separate per "bucket" (repos don't influence news and vice versa). Used to estimate the affinity of a new post.
- `liked_centroid(user_id)` ‚Üí centroid (average) of the liked embeddings; a "prior" for the user's taste (may be `None` if there are no üëç yet).
- `recall_liked(user_id, query_embedding, limit, since_days?)` ‚Üí searches ONLY the user's own üëç by similarity to the topic. The `user_id` filter is the privacy invariant ‚Äî other people's votes never show up.
- `set_focus(user_id, bucket, topic, embedding, days)` ‚Üí sets the active direction of a bucket (`repos`/`news`); replaces the previous one for that bucket (one direction per `(user, bucket)`) and sets validity (`expires_at = now + days`). `active_focus(user_id, bucket)` reads the non-expired directions for that bucket (`topic`, `embedding`, `weight`, `expires_at`). `clear_focus(user_id, bucket?)` deletes directions (for one bucket or all) and returns how many. `all_active_focus_topics()` ‚Üí DISTINCT list of topics with an active `/focus` from ANY user (used in curation to loosen the bar for those topics). Fed by `FocusItem` (see models).
- `search_pool(user_id, query_embedding, limit, max_dist?)` ‚Üí semantic search over the CURATED archive: approved + manual + what the user liked (anything good that passed curation, voted or not). Marks `liked` on what the user liked and also returns `source_platform`. Unlike `recall_liked`, which is üëç only. Runs in a transaction with `SET LOCAL hnsw.ef_search = 200`: the `approved` filter discards many neighbors from the HNSW index (the pool has a lot of rejected), so without the high `ef_search` the search would come back nearly empty. **`max_dist` is only a LOOSE candidate floor** ‚Äî this method is the 1st stage of the search; the real relevance cut is the **reranker** in `recall.py` (see `semantic_recall`). `_RELEVANCE_MAX_DIST = 0.55` survives only as the rerank-failure fallback.
- `recall_voted(user_id, query_embedding, vote?, limit, max_dist?)` ‚Üí conversational recall: posts YOU voted on, by similarity to the topic. `vote=1` üëç only, `vote=-1` üëé only, `None` any; returns `v.vote` so the bot can mark ‚ù§Ô∏è/üëé. Scoped by `user_id` = privacy. Same 1st-stage role as `search_pool`: loose `max_dist`, the fine cut is the reranker in `semantic_recall`.
- `delivered_embeddings(user_id, since_days=90, limit=800)` ‚Üí embeddings of posts ALREADY DELIVERED to this user (last `since_days`). Feeds the **repeated-news** dedup (same story from another source/day, different `source_id`) in `bot._dedup_pending`.
- `get_digest_size(user_id, bucket)` / `set_digest_size(user_id, bucket, count)` ‚Üí per-day card cap of a bucket, stored in `users.settings->digest_size` (adjustable by speech). `None` in get = app default.
- `get_balance(user_id, bucket)` / `set_balance(user_id, bucket, fresh_fraction)` ‚Üí desired fraction of NEW content per bucket, stored in `users.settings->balance->bucket` (JSONB). `None` on get = use the app default. `clear_balance(user_id, bucket?)` ‚Üí removes the mix adjustment (for one bucket, or all if `bucket` is omitted) from `users.settings->balance`, making the bucket revert to the app default (`FRESH_SLOTS`).
- `balance_signal(user_id, platforms)` ‚Üí for each voted post in this bucket, returns `(vote, delivery affinity_score)`. Feeds the auto-balancing: votes on LOW-score cards (freshness slots) vs HIGH (relevance slots) reveal whether the user prefers novelty or relevance.
- `vote_meta_signal(user_id, platforms)` ‚Üí for each voted post in this bucket (telegram origin), returns `(vote, source_platform, metadata, published_at, voted_at)`. Feeds the learning of the POPULARITY/RECENCY weights: correlates engagement (likes/RTs/stars) and age-at-vote-time with üëç/üëé.
- `get_weight_prefs(user_id, bucket)` / `set_weight_prefs(user_id, bucket, pop_mult, recency_mult)` ‚Üí LEARNED popularity and recency multipliers for the bucket (1.0 = neutral), in `users.settings->weight_pref->bucket->{pop,recency}`. `bot._auto_tune_weights` learns them once a day; `_deliver_bucket` applies them (bounded + capped so semantic dominates).

**`models.py` ‚Äî the data shapes that flow between modules**
- `IngestedPost` (dataclass): the normalized post that ANY source emits (Reddit, X, GitHub, seed, manual). Key fields: `source_platform`, `source_id` (dedup key), `source_url`, `raw_text`, plus optional `author`/`published_at`/`metadata`. It's what goes into `upsert_post`.
- `Verdict` (Pydantic): the curator's verdict, Haiku 4.5's Structured Output schema ‚Äî `verdict` (approve/reject), `confidence` (0..1), `primary_category`, `reject_reason`, `summary` (for the card) and `one_line_rationale`. Important: no numeric/string constraints in the schema (Structured Outputs doesn't support them); range validation is done in the app. It's what goes to `mark_curation`.
- `FocusItem` (Pydantic): a feed "direction" ‚Äî `bucket` (`repos`=GitHub or `news`=X+Reddit), `topic` (short topic, good for search, may be in English) and `days` (validity). Becomes a `set_focus` call.
- `ChatIntent` (Pydantic): the interpreted intent of one of the owner's chat messages ‚Äî `kind` (`steer`/`recall`/`balance`/`status`/`capacity`/`other`), `directives` (list of `FocusItem`, when steer ‚Äî each now carries a `quota`: how many bucket cards go to that topic), `recall_query`/`recall_polarity` (when recall), `balance_bucket`/`balance_fresh` and `balance_reset` (when balance ‚Äî `balance_reset=True` asks to reset the mix to DEFAULT), `status_about` (when status ‚Äî `focus`/`balance`/`both`), `capacity_bucket`/`capacity_count` (when capacity ‚Äî resize a bucket's per-day cap) and `reply` (short confirmation in PT-BR). It's the bridge between natural language and the `db.py` methods (focus/recall/balance/state/size).

**`embeddings.py` ‚Äî Voyage AI wrapper (embeddings + reranker)**
- `Embedder(settings)`: client for `voyage-4-lite` (L2-normalized vectors, 1024-dim). Exposes a public `model` (e.g. the bot uses it when saving a manual link).
- `embed_documents(texts)` (input_type `document`, for posts) and `embed_query(text)` (input_type `query`, for search queries) ‚Äî using the right type is what makes query‚Üîpost similarity work well.
- `rerank(query, documents, top_k?)` ‚Üí reorders `documents` by REAL relevance to `query` with Voyage's reranker (`rerank-2.5`, env `RERANK_MODEL`). Returns `(index, score)` sorted by score desc. It's the 2nd stage of the search (see `recall.py`): cosine distance alone doesn't separate in a single-domain archive; the reranker reads query+text together and separates for real.
- Internally it processes in batches of 32 and runs Voyage's synchronous client in a thread (`asyncio.to_thread`) so it doesn't block the event loop. The vectors produced here are exactly the ones that go into `set_embedding` and the ones the similarity queries in `db.py` consume.
- **"Back-pocket" lever (don't swap without need):** the embedder is `voyage-4-lite`. If recall regresses DESPITE the reranker, the escalation is to bump to `voyage-4` (3√ó) or `voyage-4-large` (6√ó) ‚Äî cost ‚âà free (200M tokens/account free; archive ‚âà 0.5M), but it requires RE-EMBEDDING the whole archive and pinning `output_dimension=1024` to match the `vector(1024)` schema. Deliberately deferred: the reranker already solves what the upgrade would.

**`recall.py` ‚Äî two-stage semantic search (vector recall ‚Üí rerank)**
- `semantic_recall(db, embedder, user_id, query, *, mode, vote?, limit, min_score?)` ‚Üí orchestrates the search: 1st stage pulls MANY candidates by vector (`db.search_pool` if `mode="archive"`, `db.recall_voted` if `mode="voted"`, both with a LOOSE cosine floor `CANDIDATE_MAX_DIST=0.85`, `CANDIDATE_POOL=40`); 2nd stage calls `embedder.rerank` and keeps results with score `>= RERANK_MIN_SCORE` (0.5), at most `limit`. Takes the `query` as TEXT (the reranker needs it). On rerank failure, falls back to the old distance cut (`_RELEVANCE_MAX_DIST`) ‚Äî never breaks the search. Used by `bot.cmd_search`, `bot._do_recall` and the MCP tools.
- **Why two stages:** in a single-domain archive (all AI) the vector space collapses into a cone ‚Äî cosine distance can't separate relevant from irrelevant (measured: present topic ~0.55‚Äì0.65, off-topic ~0.51 by sharing a word ‚Üí bands overlap). The reranker separates cleanly (relevant ~0.59‚Äì0.82, irrelevant ~0.29‚Äì0.46).
